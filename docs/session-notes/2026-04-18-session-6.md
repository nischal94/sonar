## Reviewer passes — what caught real bugs

**First pass (CHANGES REQUESTED, 3 P0s, 8 P1s):**

- **P0-1** — status endpoint returned only `idle` / `running` / `done`,
  never `"failed"`. Shipped the migration-006 `backfill_failed_at` column
  mid-review and wired the worker + router branch.
- **P0-2** — worker used `db.flush()` at 3 persistence points. Flushes
  never hit disk from the detached Celery session; `profile_count` and
  terminal state were being lost. Swapped to `db.commit()`.
- **P0-3** (most alarming) — worker's `if ws.backfill_used: raise` check
  directly contradicted the trigger endpoint's `backfill_used=True`
  pre-set. In production the worker would refuse the very job the
  trigger just enqueued. Tests passed only because they bypassed the
  trigger. Fix: idempotency key moved to `backfill_completed_at IS NOT NULL`.
  The trigger keeps `backfill_used` for quota semantics; the worker owns
  terminal state. **Split one overloaded flag into two columns with
  distinct meanings — this pattern is worth remembering.**

**Second pass (CHANGES REQUESTED, 1 P0, 3 P1s):**

- **P0-A** — confirmed `async_sessionmaker(expire_on_commit=False)` in
  `app/database.py`, so no lazy-reload risk inside the failure handler.
  Added inline comment at the worker. Also wrapped the failure-marker
  commit in its own try/except so a DB error there cannot mask the
  original Apify exception (original is always re-raised with full
  traceback).
- **P1-A** — `/extension/connections/bulk` response contract: expanded
  rather than changed. `upserted` still reports unique-row count;
  added `received` (client intent) and `deduped` (received - upserted)
  with safe defaults so existing callers don't break.
- **P1-C** — new test
  `test_connections_bulk_dedupes_within_single_request` covers
  last-wins dedupe on linkedin_id within a single request body.

## Surprises / judgment calls worth documenting

1. **Extension `auth_token` key** — the plan said `sonar_token` but the
   existing extension already used `auth_token`. Subagent caught this and
   matched existing convention; don't re-introduce the mismatch.

2. **LinkedIn DOM selectors in `capture-connections.js` are unverified.**
   Cannot test without a live logged-in LinkedIn session. Expect the
   first dogfood run to reveal selector fragility.

3. **`expire_on_commit=False`** is a load-bearing global setting for the
   worker's exception handler pattern. If anyone ever flips it to True
   (SQLAlchemy's default), the post-commit `ws.backfill_failed_at = ...`
   write will silently lazy-reload inside an exception handler — nasty
   to debug. Inline comment added; CLAUDE.md doesn't call it out yet.

4. **Idempotency semantics split** — `backfill_used` now means "has
   consumed the free backfill" (quota); `backfill_completed_at IS NOT NULL`
   means "has run to completion" (state machine). Admin re-run requires
   clearing both `backfill_used` and `backfill_failed_at`. There's no
   admin endpoint for this yet; a DB console is the only path.

## Dogfood checklist

Order matters — if step 2 fails, step 4 is a waste of Apify budget.

1. **Swap `.env` placeholders for real keys.** `OPENAI_API_KEY`,
   `GROQ_API_KEY`, `APIFY_API_TOKEN`, `SENDGRID_API_KEY`,
   `SENDGRID_FROM_EMAIL`. Leave Twilio/Telegram as placeholders —
   Wizard/Backfill/Dashboard don't touch them.

2. **Run the Wizard on yourself.** Enter your real capability profile.
   Watch what keywords the LLM proposes — are they too broad / too
   narrow? This feedback loop is more valuable than any synthetic
   test. `PROMPT_VERSION` on `propose_signals.py` is `v1`; bump it on
   every content change.

3. **Install `extension/` as unpacked in Chrome.** Let it observe your
   real LinkedIn feed for a few hours. Does the content script's DOM
   selector survive LinkedIn's actual markup? Does
   `capture-connections.js` scroll the virtualized connections list
   correctly? These have never run against live LinkedIn.

4. **Click "Run day-one scan."** ~$0.40 of real Apify money. First
   question: does it actually return posts? Second: do they match the
   `ApifyProfilePost` schema without parse errors? If either fails,
   pause and update `RealApifyService._ACTOR_ID` to the correct actor
   (issue #76).

5. **Live with the dashboard for 3–5 days.** Is 0.65 the right default
   threshold? Are the "matching signal phrase" summaries actually
   useful or LLM slop? Is 30s polling noisy? Is the "Draft outreach"
   button begging to be wired up before anything else matters?

**Definition of done for dogfood:** one GitHub issue per real bug,
gut-check answer to *"if this were a stranger's product, would I use
it for a second day?"* If yes → ship Discovery. If no → the issue
list tells you what to fix first.
