## Reviewer passes — what caught real bugs

**First pass (CHANGES REQUESTED, 3 P0s):**

- **P0-1** — status endpoint returned only `idle` / `running` / `done`,
  never `"failed"`. Shipped migration-006 `backfill_failed_at` column
  mid-review and wired worker + router branch.
- **P0-2** — worker used `db.flush()` at 3 persistence points. Flushes
  never hit disk from the detached Celery session; `profile_count` and
  terminal state were being lost. Swapped to `db.commit()`.
- **P0-3** (most alarming) — worker's `if ws.backfill_used: raise` check
  directly contradicted the trigger endpoint's `backfill_used=True`
  pre-set. In production the worker would refuse the very job the
  trigger just enqueued. Tests passed only because they bypassed the
  trigger. Fix: idempotency key moved to `backfill_completed_at IS NOT NULL`.
  Trigger keeps `backfill_used` for quota semantics; worker owns terminal
  state. **Splitting one overloaded flag into two columns with distinct
  meanings is the pattern worth remembering.**

**Second pass (CHANGES REQUESTED, 1 P0):**

- **P0-A** — confirmed `async_sessionmaker(expire_on_commit=False)` in
  `app/database.py` (no lazy-reload risk inside the failure handler).
  Inline comment added at the worker. Wrapped the failure-marker commit
  in its own try/except so a DB error there cannot mask the original
  Apify exception.
- **P1-A** — `/extension/connections/bulk` response contract: expanded
  rather than changed. `upserted` still reports unique-row count; added
  `received` + `deduped` with safe defaults so existing callers don't break.
- **P1-C** — new test for last-wins dedupe on linkedin_id within a
  single request body.

## Judgment calls worth remembering

1. **Extension `auth_token` key** — the plan said `sonar_token` but the
   existing extension already used `auth_token`. Subagent caught this and
   matched existing convention; don't re-introduce the mismatch.

2. **LinkedIn DOM selectors in `capture-connections.js` are unverified.**
   Cannot test without a live logged-in LinkedIn session. Expect the first
   dogfood run to reveal selector fragility.

3. **`expire_on_commit=False`** is a load-bearing global setting for the
   worker's exception handler pattern. If anyone ever flips it to True
   (SQLAlchemy's default), the post-commit `ws.backfill_failed_at = ...`
   write will silently lazy-reload inside an exception handler — nasty to
   debug. Inline comment added; `CLAUDE.md` doesn't call it out yet.

4. **Idempotency semantics split** — `backfill_used` = "has consumed the
   free backfill" (quota); `backfill_completed_at IS NOT NULL` = "has run
   to completion" (state machine). Admin re-run requires clearing both
   `backfill_used` and `backfill_failed_at`. No admin endpoint yet; DB
   console is the only path.

5. **Why we deferred P1s #2–#6, #8 from PR #77's first pass.** None
   surface 5xx in normal user flow; all are polish. Not tracked as
   individual issues because each is small enough to be absorbed the next
   time a backfill file is opened, or rolled into the dogfood issue list.
   If this turns out wrong (one of them matters in real use), that's a
   signal to tighten the review-triage bar next time.
